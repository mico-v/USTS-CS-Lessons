**五、实验结果及分析（遇到的问题与解决）**

1. 题目一分析：

* 使用 `private` 防止外部直接修改成绩，保证数据安全。
* 在 setter 中加入区间校验，非法值直接提示并拒绝写入。
* 构造重载使对象创建更灵活（默认值和完整初始化两种方式）。

2. 题目二分析：

* `totalCount` 与 `nextId` 使用 `static` 后在类层面共享，所有对象共用同一份数据。
* 若改为实例变量，每个对象只记录自己的计数，无法得到全局班级人数。

3. 题目三分析：

* `changeValue(int val)` 只修改形参副本，实参 `num` 不变。
* `changePoint(Point p)` 通过同一对象引用修改属性，实参对象状态被改变。
* `replacePoint(Point p)` 仅让形参引用副本改指向，不影响外部引用。

4. 题目四分析：

* `b = a` 是共享同一对象，故 `a == b` 为 `true`。
* `d = new Point(c.x, c.y)` 是新对象，`c == d` 为 `false`。
* 引用重新赋值只影响当前引用变量，不会反向影响原引用。

5. 题目五分析：

* 静态嵌套类 `Book` 使用 `Library.Book` 创建，不依赖外部类对象。
* 内部类 `BorrowRecord` 依赖具体 `Library` 实例，可直接访问 `libraryName`。
* 通过 `totalBooks` 验证静态成员用于全局统计是合理方案。

6. 遇到的问题与解决：

* 问题：初版容易混淆“引用传递”与“值传递”。
* 解决：通过“修改对象属性”与“形参重定向”对照实验，明确 Java 只有值传递。
* 问题：静态方法中误访问实例成员会报错。
* 解决：将实例相关逻辑放入实例方法，静态方法只处理类级数据。

**六、实验体会**

通过本次实验，我对“类是模板、对象是实例”有了更清晰的工程化理解。与只看语法相比，运行结果对比让我真正掌握了三组容易混淆的概念：

1. `static` 与非 `static`：一个面向类共享，一个面向对象个体。
2. 赋值与复制：引用赋值只是共享地址，复制对象才会产生独立实体。
3. 值传递与对象可变性：Java 始终值传递，但引用值可间接修改对象内容。

此外，封装与输入校验让我体会到“先保证数据正确，再谈业务逻辑”的设计思路。后续将继续加强代码注释、边界测试和面向对象建模能力。
